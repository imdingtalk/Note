k8s 使用kubeconfig和token访问集群

April 30, 2019

概述

kubectl命令访问集群时，默认情况下在$HOME/.kube目录下寻找名为config的配置文件，配置文件中包含集群API地址、端口号、证书等，kubectl据此建构访问集群的上下文。

场景

实际我们在使用中开发需要特定namespace的权限，并且不能有集群的远程权限，使用API访问集群，运维拥有集群的所有控制权，仅有一个集群，于是使用RBAC做权限控制。 制作kubeconfig，同时支持kubectl远程访问和通过token访问API。

环境准备

curl -s -L -o /bin/cfssl https://pkg.cfssl.org/R1.2/cfssl_linux-amd64 
curl -s -L -o /bin/cfssljson https://pkg.cfssl.org/R1.2/cfssljson_linux-amd64 
curl -s -L -o /bin/cfssl-certinfo https://pkg.cfssl.org/R1.2/cfssl-certinfo_linux-amd64 
chmod +x /bin/cfssl*

### 定义集群、用户、上下文
制作kubeconf主要有三步：

1. 为配置文件添加集群信息
1. 为配置文件添加用户信息（token访问可忽略）
1. 为配置文件添加上下文

输入以下命令，将群集详细信息添加到配置文件中：

```bash
kubectl config --kubeconfig=config-demo set-cluster development --server=https://1.2.3.4 --certificate-authority=fake-ca-file
# 例如如下：
kubectl config set-cluster kubernetes \
--certificate-authority=/etc/kubernetes/cert/ca.pem \
--embed-certs=true \
--server=https://172.16.200.100:8443 \
--kubeconfig=config-demo.kubeconfig

将用户认证详细信息添加到配置文件中（如果使用token访问可忽略此步骤）：

kubectl config --kubeconfig=config-demo set-credentials developer --client-certificate=fake-cert-file --client-key=fake-key-seefile
# 例如如下
# 设置证书认证,如果使用token认证的话，建议忽略这一步
kubectl config set-credentials devuser-sa-key \
--client-certificate=/etc/kubernetes/cert/devuser-sa.pem \
--client-key=/etc/kubernetes/cert/devuser-sa-key.pem \
--embed-certs=true \
--kubeconfig=devuser-sa.kubeconfig

将上下文详细信息添加到配置文件中：

# 设置上下文参数
kubectl config set-context kubernetes-dev \
--cluster=kubernetes \
--user=devuser- \
--namespace=dev \
--kubeconfig=config-demo.kubeconfig
# 设置上下文参数
kubectl config set-context kubernetes-test \
--cluster=kubernetes \
--user=devuser \
--namespace=test \
--kubeconfig=config-demo.kubeconfig

打开 config-demo 文件查看添加的详细信息。 也可以使用 config view 命令进行查看：kubectl config --kubeconfig=config-demo view

RoleBinding

如果我们想限制 developer用户的行为，需要使用 RBAC创建角色绑定以将该用户的行为限制在某个或某几个 namespace 空间范围内，例如：

kubectl create rolebinding developer-admin-binding --clusterrole=admin --user=developer --namespace=dev

如果有多个上下文，需要设置当前上下文：

kubectl config --kubeconfig=config-demo use-context

以上就可以使用kubeconfig文件来访问我们的集群了，其中包含的 fake-ca-file 部分均需要替换对应的自己的用户的证书文件，下回补充该部分。

新增使用token访问集群

为指定namespace分配该namespace的最高权限，这通常是在为某个用户（组织或者个人）划分了namespace之后，需要给该用户创建token登陆kubernetes dashboard或者调用kubernetes API的时候使用。每次创建了新的namespace下都会生成一个默认的token，名为default-token-xxxx。default就相当于该namespace下的一个用户，可以使用下面的命令给该用户分配该namespace的管理员权限。我们只需要给这个用户一个管理员的权限，然后使用他的token就可以了

kubectl create rolebinding $ROLEBINDING_NAME --clusterrole=admin --serviceaccount=$NAMESPACE:default --namespace=$NAMESPACE

获取指定namespace的token：

kubectl  get secret -n dev default-token-xxx

不过不建议使用默认的sa，以下使用新增的sa来访问集群

使用新增sa访问

#在指定的ns创建一个新的sa
kubectl create serviceaccount dev -n dev 
#查看相关的secret
kubectl get serviceaccounts -n dev dev  -o yaml
#查看对应的secret
kubectl get secret dev-token-1yvwg -o yaml
kubectl create rolebinding $ROLEBINDING_NAME --clusterrole=admin --serviceaccount=$NAMESPACE:$USER --namespace=$NAMESPACE
#这里列出的token是经过base64加密的，使用时候一定要解密
echo "{token}"  | base64 -d

最后的kubeconf文件应该类似这样的(仅token):

apiVersion: v1
clusters:
- cluster:
    certificate-authority-data: REDACTED
    server: https://172.16.200.100:8443
  name: kubernetes
contexts:
- context:
    cluster: kubernetes
    namespace: dev
    user: devuser-sa
  name: kubernetes-dev
- context:
    cluster: kubernetes
    namespace: test
    user: test-user-default
  name: kubernetes-test
current-context: kubernetes-test
kind: Config
preferences: {}
users:
- name: devuser-sa
  user:
    token: {这里是很长的token}
- name: test-user-default
  user:
    token: {这里是很长的token}
