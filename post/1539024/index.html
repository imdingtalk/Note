基于kubeadm搭建的etcd集群的运维实践

April 15, 2019

背景  

kubeadm提供了两种高可用方案，用户搭建k8s的高可用环境，详细对比可参考：  Options for Highly Available Topology  

堆叠的etcd拓扑

特点：

  1. etcd节点和apiserver以及scheduler，controller-manager共存，需要较小的服务器资源  
  2. 如果node出现问题，那么master节点和etcd节点都将下线
  3. 这是kubeadm中的默认拓扑。使用kubeadm init和kubeadm join时使用 experimental-control-plane 作为master 加入集群时候，在控制平面节点上自动创建本地etcd成员

外部etcd集群

特点：

  1. 需要额外的服务器搭建etcd集群，相对于堆叠型拓扑，需要双倍的服务器节点
  2. etcd节点和master分离，不像堆叠型拓扑，master或者etcd节点宕机影响更小

常用命令

#etcd需要在etcd的pod内执行
kubectl exec -it -n kube-system etcd-kube-node1  sh
#查看集群成员
ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/peer.crt \
--key=/etc/kubernetes/pki/etcd/peer.key \
member list
#健康检查
ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/peer.crt \
--key=/etc/kubernetes/pki/etcd/peer.key \
--endpoints=https://ip1:2379,https://ip2:2379,\
https://ip3:2379 \
endpoint health
#删除成员
ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/peer.crt \
--key=/etc/kubernetes/pki/etcd/peer.key \
member remove  [member ID]
#添加一个值  
ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/peer.crt \
--key=/etc/kubernetes/pki/etcd/peer.key \
put  hi  ha
#k8s删除一个节点  
kubectl delete  node kube-node3  # 但是不是删除etcd  member ，需要使用etcd的member remove  
#k8s添加节点  
kubeadm join {IP}:6443 --token 9vr73a.a8uxyaju799qwdjv --discovery-token-ca-cert-hash sha256:7c2e69131a36ae2a042a339b33381c6d0d43887e2de83720eff5359e26aec866 --experimental-control-plane --certificate-key f8902e114ef118304e561c3ecd4d0b543adc226b7a07f675f56564185ffe0c07
##参数说明
--experimental-control-plane 参数，使节点作为master加入  
--certificate-key {TOKEN}  自动从初始化节点下载相关证书到本地，如果没有记住，使用如下命令再次生成，有效期2小时  
kubeadm init phase upload-certs --experimental-upload-certs

故障演练

故障演练，模拟在默认的堆叠模式拓扑下的故障恢复能力，大致流程，put一个值->删除一个node，同时删除该node下的etcd member ->查看刚刚put值是否正常 

故障模拟

添加一对key  value ，命令参考常用命令删除一个节点，并且删除该节点的etcd  member  确保删除干净一点，把这个删除的节点的k8s信息和etcd的数据完全删除，在kube-node3上面执行  

rm  -rf /etc/kubernetes  && rm -rf /var/lib/etcd

先查看下只有两个etcd  member的时候刚刚的数据还在不在   确认两个etcdmember的时候数据正常  

故障恢复  

添加已经删除的节点  

kubeadm join xx:6443 --token 9vr73a.a8uxyaju799qwdjv --discovery-token-ca-cert-hash sha256:7c2e69131a36ae2a042a339b33381c6d0d43887e2de83720eff5359e26aec866 --experimental-control-plane --certificate-key f8902e114ef118304e561c3ecd4d0b543adc226b7a07f675f56564185ffe0c07

查看新加入的member，确认key  value 是否正常

etcd集群正常，故障正常恢复
