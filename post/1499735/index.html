MYSQL高可用架构之MHA实践

May 16, 2019

简介

MHA（Master High Availability）目前在MySQL高可用方面是一个相对成熟的解决方案

经典架构

说明：1.MHA-manager可以安装在除MYSQL-Master以外的任意服务器，唯一的要求是能和所有的DB服务器进行SSH通信2.在所有的MYSQL服务器上安装MHA-node

工作流程  

安装部署

keepalive配置

keep主要提供故障切换时候的VIP漂移  

#安装
yum install keepalived -y

配置   

global_defs {
   smtp_server 192.168.200.1
   smtp_connect_timeout 30
   router_id LVS_MASTER
}

vrrp_instance VI_110 {
    interface ens33
    virtual_router_id 110
    nopreempt
    priority 110
    advert_int 1
    authentication {
        auth_type PASS
        auth_pass keepalived
    }
    virtual_ipaddress {
        192.168.206.144
    }
}

根据 priority 字段自动判别主备，所以这个配置，在三台服务器之间应该不一致

数据库主从配置

配合5.6数据库新特性，配置主从变得异常简单，需要开启GTID，需要启用这三个参数，所有数据库添加该配置，先主库后从库，添加配置后，需要重启MySQL服务

vim /usr/local/mysql/my.cnf    
#GTID  
gtid_mode = on  
enforce_gtid_consistency = 1  
log_slave_updates   = 1

创建主从复制的用户，并在从库配置认主  

#创建主从复制用户
grant replication slave on *.* to 'repl'@'192.168.0.%' identified by 'repl';
#创建主从复制
CHANGE MASTER TO MASTER_HOST='192.168.206.129(主库ip)',MASTER_USER='repl', MASTER_PASSWORD='repl', MASTER_AUTO_POSITION=1;

MHA相关配置

各主机ssh免密码登录

ssh-keygen  #每台MySQL服务器执行，生成密钥对  
ssh-copy-id  -i root@172.17.99.x  #每台MySQL服务器执行，分发自己的公钥  
##测试免登陆
ssh ⦁	root@172.17.99.x
##免登陆配置完成

**MHA 工具部署  **

##所有节点部署MHA_node  
##centos7需要注释安装脚本中安装lib部分，用 yum install perl-DBD-mysql 安装依赖
tar zxvf mha_node.tar.gz #文件见附件  
cd  tar zxvf mha_node
bash install.sh  i  
##管理节点部署MHA_Manager，不要部署在master上  
##centos7需要注释安装脚本中安装lib和Perl库部分,用 yum install perl-DBD-mysql 安装依赖
tar zxvf mha_manager.tar.gz  
cd  tar zxvf mha_manager

修改config.ini相关配置 如下
# SQL服务器列表  
iplist="172.17.99.33 172.17.99.34 172.17.99.42"  
#绑定的vip，第一次启动需要手动绑定在master  
virtual_ip="172.17.99.43"      #所有节点机这条内容需一致  
# 绑定vip的网卡  
orig_mac_name="eth0"    #当前master对应的vip网卡名  
new_mac_name="eth0"     #高可用漂移目标主机网卡  
# mysql管理的账号，需要的权限近似root  
db_user="root"  
db_password="root"  
# mysql数据AB复制账号  
db_repl_user="repl"  
db_repl_pwd="repl"  
# 系统账号及密码  
ssh_user="root"  
# 关闭mysql事件  
db_event=0  
bash install.sh  i 
yum install perl-DBD-mysql
MHA部署完成 

服务运行检查在启动服务前，先检查配置，检查ssh免登陆配置

masterha_check_ssh --conf=/usr/local/mha/etc/app.cnf

检查主从复制状态，不报错即通过  

masterha_check_repl  --conf=/usr/local/mha/etc/app.cnf

服务运行  以上两项通过后，确认在当前的master上启动VIP，然后在MHA_ manager启动服务

masterha_manager --conf=/usr/local/mha/etc/app.cnf

**安装信息  **

#关闭：  
masterha_stop --conf=/usr/local/mha/etc/pp.cnf  
#查看状态：  
masterha_check_status --conf=/usr/local/mha/etc/app.cnf

验证方案  

#模拟主库发生故障  
#模拟前可以查看日志，查看故障切换过程  
tailf  /data/logs/mha/mha.log 
#配合5.6数据库新特性，需要开启GTID，需要启用这三个参数，所有数据库添加该配置，先主库后从库，添加配置后，需要重启MySQL服务
vim /usr/local/mysql/my.cnf    
#GTID  
gtid_mode = on  
enforce_gtid_consistency = 1  
log_slave_updates   = 1   
#重启数据库  
/etc/init.d/mysqld  restart  
#停止主数据库,命令仅供参考
/etc/init.d/mysqld  stop  
#根据日志信息，查看新的主库是否成功的启用了VIP,并确认故障的服务器已经关闭了VIP  
#成功切换后，业务上不会有任何影响，然后恢复故障的数据库  
#启动原来的主库，并同步从库同步位置  
/etc/init.d/mysqld  start    
CHANGE MASTER TO MASTER_HOST='172.17.99.x(新主库IP)',MASTER_USER='repl', MASTER_PASSWORD='repl', MASTER_AUTO_POSITION=1;  
## 要把之前的主库再次作为主库的话，可以再次模拟新的主库出现问题，将主库切换回旧主库  
##由于mha的机制，8个小时不会多次failover，所以先删除failover记录  
 rm -rf /data/mha/starcor_app/app.failover.complete  
#停止主数据库，在新的主库上操作  
/etc/init.d/mysqld  stop  
#根据日志信息，查看新的主库是否成功的启用了VIP,并确认故障的服务器已经关闭了VIP  
#验证成功
